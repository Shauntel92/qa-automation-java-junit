How can I ensure that my code, program, or software is functional and secure?
Ensuring that my code, program, or software is both functional and secure begins with writing clear requirements and translating them directly into unit tests. A program is only functional if it consistently performs the tasks it was designed for, so I focus on validating each requirement through automated JUnit tests. Each rule, such as limiting string lengths or enforcing that appointment dates cannot be in the past, is converted into both a “happy path” test and a “failure path” test. This ensures that the software behaves correctly with valid input and fails safely with invalid data. I use Maven and JaCoCo to automate builds, run tests, and measure coverage, which provides immediate feedback on whether all parts of the code are being exercised. Achieving and maintaining at least 80% coverage gives me confidence that my code meets its functional goals. To keep the code secure, I apply input validation on all fields, use clear error handling, minimize the public exposure of class members, and write explicit exception messages. I also think about what a malicious or careless user might try to do and ensure the program handles such situations gracefully. Security is built by design when validation, immutability, and least-privilege principles are followed from the start.

How do I interpret user needs and incorporate them into a program?
Interpreting user needs and incorporating them into a program requires translating vague or high-level requests into measurable, testable requirements. I start by breaking down user stories into specific acceptance criteria. For example, a user request like “allow users to enter phone numbers” becomes “accept exactly ten digits and reject any other format.” Once I have concrete rules, I map each to a test case to ensure it’s covered. Clarifying ambiguous language is key; terms like “short,” “recent,” or “valid” are replaced with exact limits or conditions. Prototyping also helps refine these needs before writing production code. Even for back-end projects, I create lightweight prototypes through test-driven development, writing a test to express the desired behavior before coding the method. This allows me to incorporate user intent early and confirm that my logic aligns with the real-world use cases. Finally, I maintain a feedback loop by reviewing test results and adjusting logic to better reflect what the end user expects.

How do I approach designing software?
When approaching software design, I now begin by clarifying scope and constraints before writing any code. I list the main domain objects, their attributes, and their rules—for example, a contact has a unique ID, name, phone number, and address, each with length or format restrictions. I then design small, cohesive classes that model real-world entities and define how services interact with them. Each service has a single responsibility, such as managing contacts or appointments, and methods are designed to be simple and predictable. I capture all validation in constructors or setters so invalid objects cannot exist in memory. I plan the testing strategy before coding, ensuring every function and rule will have at least one associated unit test. I also follow the Red-Green-Refactor cycle of test-driven development: write a failing test, make it pass, and then refactor for clarity. By focusing on small, self-contained modules, I create code that’s easier to test, maintain, and scale.
Designing software also means thinking about the long-term maintainability of the project. I’ve learned that disciplined practices—such as keeping classes small, writing descriptive names, and avoiding unnecessary complexity are what prevent technical debt. A consistent testing and refactoring process ensures that as the system grows, it remains reliable and secure. I’ve also learned to appreciate the connection between functional correctness and design simplicity. Clean, modular code is easier to test and debug, while over-engineered systems introduce risk and instability. This course taught me that good design is not about writing more code, it’s about writing code that is intentional, testable, and adaptable to future changes.
